GDPR - Process-Controller Contracts

Largely, no Blueberry company should be in the business of processing personal data for another controller. Rather, Blueberry Systems and Blueberry Consultants build software that assist our customers in processing their own data. But they will have some incidental processing of personal data for another controller during, e.g. database analysis, trouble-shooting / debugging, or pseudonymisation / anonymisation work.

All Blueberry companies process data for their own purposes, but that doesn't require a contract.

Eventually, the Information Commissioner's Office will start pushing for contracts to be standardised, and/or for contracts to be certified by some separate organisation. But those standards and structures do not yet exist.

Below is a template contract that you can use for Software Development and IT Systems Support processing. It must not be used for server hosting. It must not be used when we're sending unanonymised personal data overseas. Please do replace Blueberry Consultants Ltd with Blueberry Systems Ltd where appropriate.

You’ll also need to fill in a Registry Form for Blueberry’s own records. And when we’ve got a signed contract you’ll need to attach it to your PTS project. There’s a new tab: “Processor-Controller Contracts” for that.

More generally, the GDPR requires processor-controller contracts to contain the following information:

  • Purpose of the processing, e.g. staff administration
  • The expected duration of the processing
  • Categories of data: See the green box on GDPR - Privacy Notice
  • Categories of data subjects: e.g. employees

And it requires the following terms:

  • The processor will only act on the written instructions of the controller (unless required by law to act without such instructions);
  • The processor will ensure that people processing the data are subject to a duty of confidence;
  • The processor will take appropriate measures to ensure the security of processing;
  • The processor will only engage a sub-processor with the prior consent of the data controller and a written contract;
  • The processor will assist the data controller in providing subject access and allowing data subjects to exercise their rights under the GDPR;
  • The processor will assist the data controller in meeting its GDPR obligations in relation to the security of processing, the notification of personal data breaches and data protection impact assessments;
  • The processor will delete or return all personal data to the controller as requested at the end of the contract; and
  • The processor will submit to audits and inspections, provide the controller with whatever information it needs to ensure that they are both meeting their Article 28 obligations, and tell the controller immediately if it is asked to do something infringing the GDPR or other data protection law of the EU or a member state.

GDPR - How To Handle Customer's Data

Project managers who are processing unanonymised personal data need to be aware of the following. 

  • By receiving the data Blueberry counts as processing it. That includes receiving a file, receiving a database, remote login to see the data on a customer's system, or even viewing the data across the internet in a web browser.
  • "Personal" data means "any information relating to an identifiable person". Even if the data was received in a commercial context it remains personal data.
  • Blueberry shouldn't be in the business of processing personal data for customers except for: database analysis, trouble-shooting, hosting it on a server, anonymising / pseudonymising. Pseudonymising means that the personal data is currently not relatable to an identifiable person, but with the addition of a little more data their identity could be recovered. We might prefer to give the customer tools to anonymise / pseudonymise rather than perform that processing ourselves.
  • Customers should send Blueberry a contract for that processing. Contracts should be recorded on the Project Documents tab in PTS.
  • Blueberry doesn't seem to have a legal duty to remind customers that they should send such a contract.
  • We should only process based on the written instructions of the customer. Typically the contract, but an email may suffice. If the instructions are for ongoing processing, then we really ought to have it in a contract.
  • We should not send personal data to our offshore staff.
  • We must not use a sub-processor organisation without the prior written authorisation of the customer.
  • When asked, we must co-operate with supervisory authorities (such as the UK's Information Commissioner's Office)
  • We need to ensure the security of the processing
    • Staff training
      • Make sure everyone who works with the personal data has read GDPR - Information Security Policy
      • Make sure the PMs know what to do when something has gone wrong
    • Physical security - Details of Physical Security at Blueberry
    • System security – the security of your network and information systems
    • Data security –
      • Access controls (do only your team and sysadmins have access?),
      • Authentication (is it really the right person logging in?),
      • Audit (can you see what has been done?)
    • Online security – the security of the hosting services, logging services, mail services, etc. You may find some of the answers at GDPR - What customers have asked helpful for determining what security we typically use.
    • Device security –
      • Bring-your-own-Device policies,
      • Encryption
  • Of the many things that most processors would have to ensure, only a few are typically applicable to Blueberry's cases. Each of these are "where reasonable bearing in mind the costs". If you're doing something other than database analysis, troubleshooting, or anonymisation on personal data, then please see Article 32 / ask Duncan. Below, the ones that we're typically required to ensure:
    • confidentiality
    • encryption
    • pseudonymisation or anonymisation
    • if ongoing: a process for regularly testing, assessing and evaluating security
  • We must keep records of its processing activities (we will have a processing register for this, separate from the asset register)
  • We must notify any personal data breaches to the customer, without undue delay. That includes, but isn't limited to the following. Feel free to check with Duncan if you don't know whether a breach should be reported:
    • access by an unauthorised third party;
    • sending personal data to an incorrect recipient;
    • computing devices containing personal data being lost or stolen;
    • alteration of personal data without permission; and
    • loss of availability of personal data.
  • Ask Duncan to fill in more detailed advice if/when the UK is outside the European Union
