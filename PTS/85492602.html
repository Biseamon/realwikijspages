Blueberry Group : Tasks, Issues and Tickets

Created by Jason Cozza on Aug 24, 2020

Items entered into PTS will be one of the following:

  • Issues – Bugs and other problems that need to be fixed.
  • Tasks – Items of developer work that need to be completed, such as new features, etc. These are typically chargeable items linked to work that is planned in a Project.
  • Tickets – Are given to customers and non-Blueberry personnel so that they can track an issue or task that they have reported. When the issue or task has been completed, the ticket holder can be notified. Several tickets can correspond to a single task or issue.

If you select any of these options from the menu bar, you will be given the option of viewing all of the current or to open a new issue / task / ticket.

Whenever one of these items are modified or changing, all members of the issue team will be sent an email informing them of what changes have been made.

Viewing

When 'All...' is selected from the relevant menu, you will be given a table with all of existing entries in it. The list of items can be narrowed down by opening the Show Filters panel to quickly assign filters. Alternatively, you can type something into the search text field and narrow the list by keywords.


Any item of interest can be opened by clicking on either its number or title.

Alternatively, if you already know the item number for a task or issue that you want to view, you can type the item number into the PTS search box.

Adding a Task, Issue Or Ticket

When creating a new issue / task / ticket, simply enter all of the relevant details and then select the Update Issue/Task/Ticket button.

If you wish to add an attachment, this cannot be done until the item has already been saved and allocated a number. Once the item has been saved for the first time (as above), simply add the attachment and click the Update Issue/Task/Ticket button again.

There are a number of fields that can be filled in but the most commonly used ones are:

  • Task - the parent task that this item me need to be attached to.
  • Subject - a very brief summary of the item.
  • Description - a more detailed explanation of the item with the option to include rich text.
  • Attachments - any files that may be associated with the item (Note - attachments cannot be added until the issue has been created).
  • Issue Team - everyone in the list will be emailed whenever the item is updated (add memebrs by selecting them from the drop down menu and then pressing the add button).
  • Priority - an indication of how urgently the item needs looking at. Unless a manager says otherwise, tasks and issues should be worked on in priority order - Urgent tasks / issues first, then High, then Normal and Low tasks/ issues last.
  • Status - an indication as to who currently needs to be looking at the item.

Once all necessary fields have been completed, click 'Update Issue/Task/Ticket' to save.

If the task or issue is successfully saved, all members of the team will receive a summary email.

  • Active - The task / issue has been opened by a manager or tester and is currently being worked on by a developer. If a developer changes the status of a task or issue to Active, it lets a manager know that that item is being worked on.
  • Waiting for Developer Team (WfDT) - A manager has assigned the task / issue to the developer team to complete. These issues should be completed in priority order. When work starts on a Waiting for Developer Team item, the status should be changed to Active.
  • Waiting for Manager / Tester - A developer has a comment or needs more information from the manager or tester. This is useful if a developer has a question, is waiting for more information or does not agree with a task or issue. When changing to this status, the developer should make sure that a comment is added.
  • Waiting for Client - The task / issue cannot be progress until further information or content is received from the client.
  • Waiting for Third Party - The task / issue cannot be progress until further information or content is received from a third party other than the client.
  • Waiting for release to testers (WfRtT) - Code changes for the task / issue has been made but a new build has not yet been provided for testers or managers to test. If changes are committed to SVN, the status is automatically updated to this. Details on committing to SVN are available here .
  • Waiting for Test (WfT) - A new build containing the code for the task / issue is ready and waiting for a tester or manager to test. When the build is available, the developer should update all relevant tasks or issue to Waiting for Test.
  • Waiting for release to customer - The task / issue has been completed by the developer and verified by a manager or tester but has not yet been released to customers.
  • Closed - The task / issue has been completed by the developer and verified by a manager or tester and no further action is required. Generally only managers and testers will close tasks or issues.

My Working Items

Any task / issue that are currently assigned to you will appear on the My Working Items screen

The items displayed can be filtered by Status.

Generally speaking, assigned items should be completed in priority order - Urgent items first, then High, Normal, and Low last. Of course, if a manager asks for items to be completed in a different order then they should be completed in the order that he/she specifies.

Estimates

One of the big problems with remote work is that it can be quite easy for people to be confused about the size of a task and progress within it. A project manager might ask for an item of work that he thinks will take a week’s effort and the developer might think it will take three weeks.

Sometimes the reason will be that the project manager isn’t aware of a technical difficulty. Sometimes the developer will not be aware that there’s some existing work that can be copied. Sometimes the developer and the project manager will have entirely different ideas about what the client wants to see.

The worst time to find out about the differences of opinion is when the project manager asks “Why has this taken twice as long as I expect?” It is particularly bad if it turns out the answer is “Half of what’s been implemented isn’t wanted by the client”.

To avoid that situation we encourage developers to do two things:
 

  • Add a PTS estimate for each task, or issue, that looks like it might take more than a few hours.
  •   If a task is expected to take more than 3 days, then break it down into smaller tasks that are not expected to take more than 3 days.
  •   Most task descriptions should be something that one of our testers can read and say “yes, that’s complete” or “no, that’s incomplete”. It’s absolutely permissible to do that by referencing a design document.


By doing that, it becomes much easier for a project manager to spot a potential problem early and save developers from unnecessary work.

Updating a Task, Issue Or Ticket

Updating a task or issue is done in exactly the same way as when a task or issue is added. To start editing, just find the relevant task or issue on the All Tasks / All Issue screens and then click on it to open it for editing. When editing is complete, click Update Task / Update Issue.

Alternatively, when a commit is made to SVN, by entering the working item number in SVN, the relevant Task or Issue will automatically be updated in PTS - a comment will be added and the status will be updated to 'Waiting for release to testers'.

When changes to a task or issue are saved, all members of the team will receive a summary email, indicating what the changes are.

Updating a Task or Issue by E-mail

Instead of using the PTS web app to create and edit tasks/issues/issues, you can simply send and email with special tags at the end instead. This is useful if someone reports an issue or task to you by email and you quickly want to open a task/issue in PTS.

To find the correct email address to send the email to you first need to know the short code for the project. Then you need to decide if you want to open an issue, task or ticket and then. Once you have these two pieces of information, the email is formed as follows:

<task/issue/ticket>-<customer short name>-<project short code>@pts.bbconsult.co.uk


When the email is sent, the subject of the email will be used as the Subject in PTS and the body of the email will be used as the item Description. All other fields need to be configured using ## tags at the end of your email.
