What is the difference between Tasks, Issues and Tickets? 

Items entered into PTS will be one of the following: Issue, Task or Ticket.

Tasks and issues are very similar in PTS - they have a lot of common features and are, essentially, the same thing inside the PTS database - but should be used in different situations as explained below: 

  • Issues - Bugs and other problems that need to be fixed.

     
  • Tasks - Items of developer work that need to be completed, such as new features, etc. These are typically chargeable items and are linked to a plan of work within a Project.

     
  • Tickets - Are given to customers and non-Blueberry personnel so that they can track an issue or task that they have reported. When the issue or task has been completed, the ticket holder can be notified. Several tickets can correspond to a single task or issue.

Can I update many tasks at once? 

The 'Status' and 'Assigned To' fields can be updated for many tasks at once. To do this:

  1. Find the Tasks on the All Tasks page.
  2. Check the checkboxes next to the relevant Tasks.
  3. Go to the bottom of the page.
  4. Select the relevant Status or Assigned To value from the drop down menus available
  5. For changes to Assigned To, click the Set Assign button to save the changes and the for changes to the Status field, click the Release button


How do I give a Customer access to the Portal? 

A customer user can be given access to either all projects for the customer company or just specific projects.

To add a user to all projects belonging to the customer company:

  1. Go to Projects -> Manage Customers.
  2. Find and click on the customer
  3. Go to the Customer Contacts tab
  4. Enter the name of the contact in the Contact field. If the user already exists, click Add. Otherwise click the + icon and complete the user's details before clicking the Add button.
  5. Click Save
  6. Go to Admin - Manage Users
  7. Find the user and click Edit
  8. Set a password on the Basic Details tab and add the Customer Portal role on the Groups, Roles and Permissions tab
  9. Click Save

     

To add a user to a specific project belonging to the customer company:

  1. Go to Projects - > All Projects
  2. Find and click on the project.
  3. Go to the Customer Contacts tab
  4. Enter the name of the contact in the Contact field. If the user already exists, click Add. Otherwise click the + icon and complete the user's details before clicking the Add button
  5. Click Save
  6. Go to Admin - Manage Users
  7. Find the user and click Edit
  8. Set a password on the Basic Details tab and add the Customer Portal role on the Groups, Roles and Permissions tab
  9. Click Save

Note - Instead of setting a password for the customer, a customer can set their own password by using the Lost Password option on the login screen to request a password reset email.

Uploading the XLS File

 

To upload the tasks in PTS:

  1. Login to PTS
  2. Go to the All Tasks page for the appropriate project
  3. Click the 'Import' button
  4. Select the xls file
  5. Click the 'Send Stats Report' checkbox - this will notify you when the upload is complete and of any errors that have occurred during upload.
  6. Click OK

Alternatively, to upload the tasks by email attach the xls to an email (note - only one attachment per email is supported) and send the email to import-task@pts.bbconsult.co.uk or import-issue@pts.bbconsult.co.uk. The email should not contain anything apart from valid PTS commands.

 

What reports are available from PTS? 

PTS contains a number of different reports. These are detailed below.

Project Task Progress

This report allows you to view the tasks (and optionally the Issues too) for a Project. The time spent on each task can be compared to the estimated time. Tasks are sorted by parent tasks, child tasks and issues (moving from left to right across the first 3 columns). Tasks can also be grouped by Scheduled. An overall summary is given at the bottom of the report.

How can a Manager change a Developer's working hours? 

A developer's expected working hours and working mode (ie full time or part time), are set on their employee page in PTS under the 'Billing Details' tab.

If the developers working mode is set to 'Full Time', the working hours will always be set to 40 hours and the box cannot be edited to change this.

However, if the the developer is 'Part Time', the working hours box can be edited to manually set how many hours they are expected to work.

How can a Manager change a Developer's time owed? 

Each developer is expected to work a certain amount of hours. The number of hours expected can vary from developer to developer so this value is specified for each developer on their Billing Details page in PTS. If the developers working mode is set to ‘Full Time’, the working hours will always be set to 40 hours and the box cannot be edited to change this. However, if the the developer is ‘Part Time’, the working hours box can be edited to manually set how many hours they are expected to work.

 


However, rather than rigidly forcing a developer to work those exact hours each week, they are allowed be flexible with their working hours. For example, if they work a few extra hours this week, they can work a few weeks less next week and vice versa.

To ensure that a developer is not consistently working too few or too many hours tracks, PTS tracks that developer's 'Hours owed' - this is the difference between their expected working hours and the number of hours booked on their timesheets. A positive figure means the developer needs to do a bit more work, whilst a negative figure means the developer has worked a bit too much.

The Hours owed is tracked on a rolling 4 week basis with the developer's manager receiving an email if the value becomes too high in either direction.

 


The example above shows that the developer owed 20 hours from before the start of the current 4 week window but then worked 130 hours more than expected during the 4 weeks. As a result he is now owed 110 hours.

If a manager receives one of these emails, they should check that the developer is not working too many or too few hours. If the manager is satisfied with the situation (if the hours are as a result of an error in PTS, for example), they can simply reset the developer's owed hours by editing the Hours owed field in PTS and replacing the current value with 0.

The manager who will receive the email is determined by the developers Main Project setting which is specified on their Working Items page on PTS. The manager of their Main Project will be inherited as their manager. If no Main Project is set for any reason, the Developer's Manager field is used instead to manually choose a manager (Note - if a Main Project has been selected, the Developer's Manager field is disabled)
