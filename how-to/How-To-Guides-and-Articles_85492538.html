Contacting System Administrators 

 

For urgent/critical issues or tasks, contact sysadmins via email at sysadmins@bbconsult.co.uk.

A non-exhaustive list of things which sysadmins consider urgent:

  • System failure (production system: database, infrastructure, server, etc.).
  • Overdue task.

If your task does not fit into the above, continue with this guide.

Be reasonable about task deadlines. For example, if you've decided to submit a task on the day that it needs to be completed when it could have been submitted a week prior, please don't be offended if a sysadmin says that it can't be prioritised. Simply setting something as urgent in PTS won't make it move any quicker, and if it's truly urgent you should be using email.

Sysadmins appreciate that customers can make things difficult with regards to scheduling, but do try to leave as much time for completion as possible.

Do consider whether the task you are submitting is actually for sysadmins. For example, if it's an issue with PTS you should contact Carl Smith or Oliver Goodman.

Step-by-step guide

  1. Open PTS
     
  2. Click the Contact Sys Admins button at the top-right corner of any page.
     
  1. Fill out the form as desired.
    If your work is project specific, then remember to select it.
    Try to include as much information as possible in the Further supporting information field to reduce the time taken for work to begin on the task.
    You can suggest who should take up the task by mentioning them in the Further supporting information field.
  2. Click Send.

 

Alternatively, you can create a task/issue/ticket as normal in PTS and assign the Sys Admins group to ensure they receive it.

Key points

Sys admins will automatically be informed of the new task and will delegate it internally.

When they want you to respond to the task, they will set the status to Waiting for End-User.

When you've responded and want them to respond or continue work, set the status to Waiting for Manager/Tester or Waiting for Developer team.

Respond to tasks as quickly as possible so things can move quicker.

Don't split discussion of a PTS task into email unless it has evolved into an urgent task. Sysadmins will simply direct you to use the PTS task if you try this.

If you feel that your task has been ignored (no responsible set, or no initial follow up), do send out a reminder to sysadmins@bbconsult.co.uk.

Try to keep the task on-topic. If the task drifts away from the original topic and the task requirements were fulfilled, sysadmins will likely close it as completed so you should create a new task with new requirements if required.

BlueSprints (Refresher for Blueberry Systems PMs currently) 

 

 

 How blue am I? Here’s your checklist! Here are a few BlueSprints things you should be doing, and this now applies to all projects, except those that are 95% completed e.g. MCA Control Room, Where’s Free.
 
USE LEVEL 1 TO LEVEL 3 TASKS
 
Your Task Tree in PTS should have three levels. Level 1 Tasks are the customer’s requirements – the “what”, Level 2 Tasks are your design response to each requirement – the “how”. There may be one or more Level 2 Tasks for each Level 1. Level 3 Tasks are the work instructions for the developers. There may be one or more Level 3 Tasks per Level 2.
 
MAKE YOUR REQUIREMENTS “USER STORIES”
 
Level 1 Tasks are your requirements and they should typically be expressed in customer user language. Ideally they will be written by the customer, but sometimes they will be the result of a workshop session with the customer. Nevertheless, they should be in a format such as “I want a page where I can view and update information about any of my suppliers” or perhaps “The system must have a page for listing and editing suppliers”. The exact format is flexible but each requirement should be atomic, clear and complete.
 
IDENTIFY YOUR PRODUCT OWNER
 
You should identify which one person within the customer organisation owns the requirements. This is the Product Owner. It may or may not be the same person as the customer’s Project Manager. Really, it needs to be the person who the system is for and who can make executive decisions about what they want from the software. Examples would be Cliff at Uvex, Amelia at EE, David at Hudson, ? at RCO (interesting one – forces the customer to decide who is ultimately in charge of the spec!). You can set the Product Owner role in PTS the customer contact grid. Once set up, this person can see Tasks for approval in the Portal.
 
GET LEVEL 1 AND LEVEL 2 TASKS APPROVED BY THE CUSTOMER IN THE PORTAL
 
Firstly, make sure that you set up your customer with Portal access at the start of the project. This is a startup activity. As you create your requirements (Level 1 Tasks) and design (Level 2 Tasks) you should set the status to “WFCA” (Waiting For Customer Approval). These Tasks will then appear to the customer’s Product Owner in the Portal, where they can add comments and approve them.
 
You don’t need to produce a spec document – unless the customer explicitly wants one, in which case we have a document template that can be used where an export of the Level 1 and 2 Tasks from PTS can be inserted.
 
MAKE A RISK LOG – AND CHECK IT WEEKLY!
 
Every project should have a Risk Log created at the start of the project and checked/updated weekly. We will be introducing a tool called PRISM soon to help you tease out where the project risks are. Remember that a Risk Log is totally useless unless you make an action plan to mitigate each risk – and review the Log and Actions weekly.
 
CREATE A DEVELOPMENT CATALOGUE
 
Once your Level 1 and Level 2 Tasks are approved by the Customer, you can create the corresponding Level 3 Tasks. These are the actual work packages for the developers. Each L3 Task should be small (max 2 days work) and well defined. You should agree an estimate with the developer that you are both happy with, erring on the pessimistic side. Each of these Tasks should then be set to status DCAT (Development Catalogue) so they are ready to be allocated to a Sprint.
 
PLAN OUT YOUR SPRINTS
 
Once your Development Catalogue is complete, it is ideal to plan out ALL of your Sprints. Of course, Sprints will change and new Sprints will be added as the project progresses, but by starting off with an initial set of planned Sprints, including all DCAT Tasks, you are effectively creating a project plan for the development work.
 
SET UP SPRINTS WITH A FIXED DURATION AND LEAVE 20% CONTINGENCY
 
Setting up a Sprint has these steps:-

  1. Decide the duration. The duration must be no more than 3 weeks. 1 or 2 weeks should be normal.
  2. Identify what developer resources you have and their availability, taking into account holidays and their involvement in other work. This gives you a number of available developer hours.
  3. Set at least 20% contingency in your Sprint.
  4. Allocate Tasks from your Development Catalogue to the Sprint until all the available time is used (excluding the contingency).


 
Example:-

  1. My Sprint will be 2 weeks (10 working days)
  2. I have two developers. There is a Russian bank holiday within the Sprint and one of my devs is needed for two days on another project. So that gives me 9 days for one developer, and 7 for the other. Total 16 days – 128 hours.
  3. I set contingency at 20% so 26 hours is not available for allocating – it’s there for fixing problems, handling overruns etc.
  4. I allocate Tasks to the Sprint to fill up the remaining 102 hours based on estimates I agreed with my devs.


 
HAVE A DAILY REVIEW WITH YOUR DEVS AND UPDATE YOUR ESTIMATES
 
Each morning, talk to your devs on Skype. Focusing on the Tasks they are working on, ask them “What did you do yesterday?”, “What are you planning to do today?”, “Is the Task on target or are there any impediments requiring a re-estimate?”. Record progress against your estimates and this will feed into your Burndown Chart. Currently, we are using a spreadsheet for this but hopefully it will soon be possible to do this in PTS.
 
USE BURNDOWN CHARTS AND SHOW THEM IN THE TEAM STANDUP
 
By updating estimates with the devs every day, you will update your Burndown Chart. Show this in your Team Standup when reporting project progress on Monday, Wednesday and Friday. Also, you can show it to your customer unless there is a good reason not to. In the future, it will be made available to the customer through the Portal.

Configuring Xcode with SmartSVN 

 

These guides were written using a Mac Mini running OSX 10.8.5, Xcode 5.1 and SmartSVN 7.6.

Configuring SmartSVN

  1. Download and run SmartSVN 7.6 or later
  2. Agree to the license terms and click continue
  3. Select which edition you have. If registered then browse computer for your license file and click continue
  4. Choose ‘My repositories are already set up’ and press finish
  5. Choose ‘Check out project from repository’ then OK
  6. Type the link to the repository then continue
    For example https://svn.bbconsult.co.uk/repository/ampetronic_fsm
  7. Enter your username and password then login
  8. Once the directories have loaded press continue
  9. Choose where the files will be stored by clicking the folder next to the local directory then click continue and finish

Configuring Xcode

  1. Load Xcode and at the bottom click ‘Open other’
  2. Locate the local directory for your SVN folder
  3. Find the project in trunk
  4. Open ‘YOURAPP.xcodeproj’
  5. ‘Ignore’ or  ‘Check Out..’ additional working copies (depends on if you have access to them)
  6. Go to Xcode > Preferences and press the ‘+’ at the bottom
  7. Add your Apple ID as an account
    Note: Will not work if;
    – Apple ID is not in the Developer Program
    – Apple ID is not part of the team
  8. Send a request for a Certificate, once accepted double click on ‘Blueberry Consultants Ltd’
  9. Press Refresh and then Done
  10. Click on the Xcode Project on the left bar in the Xcode Builder
  11. Go to Build Settings, under Code Signing  > Code Signing Identity choose your certificate for all code signings
  12. Under Code Signing  > Provisioning Profile choose ‘Internal_Distribution’
  13. Run the app in the simulator or a device

     

Note:
