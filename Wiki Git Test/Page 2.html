Blueberry Group : BBWT3 - Login Page & Reset Password

Created by Jason Cozza on Sep 21, 2020

Overview

During login, we show an on-screen indication of password strength. Passwords are not stored in the database. We use the ASP.NET Core User Management feature to store strong hashes of passwords using the PBKDF2 algorithm. Passwords are never known to anyone except the user. Passwords are checked client side against the "Low-Budget Password Strength Estimation". Additionally password acceptability can also be determined by the server by which characters are used (lowercase, uppercase, digit and symbol), as well as specifying the minimum password length.

  • Present a user with an email and password form to fill out.
  • Present the option to create an account.
  • Present the option to recover an account with a forgotten password.
  • On creating an account it will check the password that the password meets the specified requirements:
    • The password is at least x characters long (admin configurable).
    • That the password is sufficiently secure.
    • If password is to easy to crack then it will not allow the user to sign up with that password, and will offer a suggestion on how to strengthen the password; "Add another word or two. Uncommon words are better", and to avoid using sequences.
    • On a successful account creating user will be logged in after completing account creation.

    • Has the ability (if enabled) to check submitted passwords against the HIBP (Have I Been Pwnd) API.

  • For a forgotten password:
    • Once the user's email has been entered and has clicked "Send", a pop up message will appear confirming the action.
    • A user then clicks the link sent to enter in a new password.
    • After entering a new password, a pop up message will be shown stating a successful change with a link to log in.
    • If user sends forgotten email twice and then clicks on the link from the first time, site displays an "Invalid token" error.
    • Password reset email contains the IP of the person who initiated it.

    • Not currently implemented Task:179316
      Does not reveal information if user is within database. When an email is submitted it will confirm an email has been sent regardless of whether user is within database or not

  • On failed login:
    • If user not in database - does not reveal that user is not in database, gives message saying username/password combination is invalid.

    • Not currently implemented Task: 179318

      Current implementation allows for an attacker to gain information on whether a user is registered, by entering in username:password incorrectly 3 times and triggering a 10 second time out; To avoid this issue a time out should be made after 3 unsuccessful attempts regardless of if the user is registered or not.

Test Case

  1. New Account Creation:
    1. User goes to account creation page.
    2. User enters in details and password.
    3. The password is checked against requirements.
    4. If it passes requirements, then account is created.
    5. User's password is then checked against the HIBP database if enabled.
    6. A new pop up will display stating whether or not the password the user entered is on the database.
    7. If it is on the database it will show how many times it has come up, and will warn the user not to use that password.

  2. Forgotten Password:
    1. User has forgotten password.
    2. User enters in email address used to sign up.
    3. BBWT3 sends a password reset email.
    4. User clicks on the time-expiring password reset link.
    5. User enters in new password.
    6. User logs in with new password.

For Testers

  • For Login:
    1. Go to BBWT3 URL.
    2. On screen will be displayed a form for username and password.
    3. Enter details and click "Log In".
  • For Forgotten Password:
    1. Go to BBWT3 URL.
    2. On the screen click on the "recover a forgotten password" hyperlink.
    3. On screen will be displayed a form for entering in your email address.
    4. Enter details and click "Send".
    5. Log into your email account.
    6. Click on the BBWT3 link to reset password.
    7. Enter new password.
    8. Click on login and fill out email and password.

For Project Managers

  • Having a simple login and forgotten password functionality is built directly into BBWT3.
  • All passwords are stored in the form of PBKDF2 hashes.
  • Password hashes are compared against known leaked passwords.
  • Any user who attempts to use a weak or known leaked password is notified and asked to give a different one.
  • Passwords on new accounts can be checked against the HIBP database. If password is found on the database it will inform the user that the password is not secure.
  • Silently fails if API is unreachable, meaning that the web application will not go offline if a 3rd parties does.
  • Ability to utilise further security precautions where necessary with Two Factor Authentication.

  • To be implemented: Password Reset requires that an email be sent to the user with a time-expiring link. The user uses this link to reset their password (31-01-19).

  • To be implemented: If a user changes their email address, we send an email to the previous address to confirm the change, and note in the UI that the email is not verified until the link is clicked (31-01-19).
  • Note: Does not consider the use of alt codes to be secure (31-01-19).
  • Note: Database stores username and email as it was entered. Does not change capitalisation - no rules regarding this (31-01-19).

For Marketing

  • In built login and forgotten password functionality.

  • Does not store the password but rather a cryptographically strong hash using state of the art algorithms.

  • Checks hash against known leaked passwords and if there is a match it will ask the user to use a different password.

  • Checks hash against HIBP database to see if it has been compromised and if it has, it will warn the user never to use that password.
