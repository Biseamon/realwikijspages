Blueberry Group : A9:2017 - Using Components with Known Vulnerabilities

Created by Jason Cozza on Sep 22, 2020

Overview

Components, including libraries and frameworks, may be taken from the open source community and should be used with caution in case vulnerabilities are lurking. As a vulnerable component is exploited, attackers can leverage it and cause the app serious damage and a major loss of data that can undermine the app, and perhaps even the organization.

By using components with known vulnerabilities, attackers can take advantage of that using SQLi and an XSS (among other attack methods) to attempt to takeover and occupy the app.

Example

Components typically run with the same privileges as the application itself, so flaws in any component can result in serious impact.

Such flaws can be accidental (e.g. coding error) or intentional (e.g. backdoor in component).

If the web application uses undocumented dependencies this could lead to them not being updated and thus pose a security risk.


How BBWT3 Mitigates This Vulnerability

  • Constant testing with 3rd party tools

BBWT3 uses a tool known as Acunetix which scans our code for potential vulnerabilities. When one is identified it is quickly patched so that the integrity of BBWT3 remains.

High risk sites should be tested regularly with both Acunetix (for application vulnerabilities) and OpenVAS (for system vulnerabilities). Note well: Hosts such as Amazon require notification up-front of vulnerability scans. Otherwise, they're unable to differentiate between real and simulated attacks.
