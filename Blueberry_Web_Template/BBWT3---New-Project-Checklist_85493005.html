New Project Checklist 

 

Project Managers

When creating a new BBWT3 project, follow these steps:

  1. Ensure that a PTS project exists, otherwise create one
  2. Decide which BBWT3 version to use:
    1. Latest stable branch (recommended)
    2. Current development branch (higher risk)
  3. Raise a PTS task with Sysadmins asking them to: Create a new Gitlab project by forking BBWT3 code. Reference which branch you require.

When creating a new BBWT3 based project, run through the following checklist:

  • Fork the project into the Blueberry namespace (this might require you to fork it into a private namespace, rename, then transfer to the blueberry namespace).
  • Create any required infrastructure for the project.
    • What kind of deployment target is required? Is it a single server or load balanced? Will the DB be on the same server as the application or on RDS?
    • If using a single server, consider using the latest BBWT3 based AMI which has a database ready to use.
  • Prepare any AWS keys that the application requires, e.g:
    • IAM user with S3/SES permissions
  • Disable CI/CD in the project settings until the CI file has been updated for the project.
  • Adjust the CI/CD as appropriate, to be able to deploy to any infrastructure you've created for this project.
  • Adjust the SonarQube configuration in the repository:
    • The key SONARQUBE_KEY in gitlab-ci.yml should be changed so that is a new key, not yet used on SonarQube.
    • The property sonar.projectName in SonarQube.Analysis.xml be changed so that is a new name, not yet used on SonarQube.
  • Store any secrets (logins, passwords, etc.) in BitWarden.
  • Add any secrets to the CI/CD variables settings for the project, and reference them correctly in the CI file.
  • Once the CI file is ready, re-enable CI/CD in the project settings.
