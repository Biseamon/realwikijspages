Blueberry Group : BBWT3 - Logging 

Created by Jason Cozza on Sep 21, 2020

Overview

Logging system uses the 3rd party software called Graylog.

  • Logs current user ID.
  • Logs any impersonation currently in place.
  • Logs any "real user name" defined on current user.

There are 3 logging destinations available by BBWT3:

  • Log error file.
  • Greylog.
  • Database.

File logging should not typically be used in live or test environments, and should be considered for development machines only. BBWT3 principles discourage writing to files on the webserver, and discourage developers connecting to the webserver to read files.

Greylog should be the preferred solution as it can accept application logging in addition to logging from other sources, e.g. database, OS, IIS.

Test Case

  1. Web application is being actively tested by QA team.
  2. A tester causes an error.
  3. This error can be diagnosed due to the logging from Graylog.

Developer Notes

During development dbdoc should not need audit logging in addition to the repository. Any work that goes through the metadata table is already being tracked in the source code repository.
Within Serilog you are able to set the minimum level of logging per source. Within the configuration you need to have the section “Override” inside the “MinimumLevel”. For more information and an example see here.

For Testers

Go to https://graylog.bbconsult.co.uk.

  1. Enter in LDAP username and password.

For Project Managers

  • Able to quickly identify issues. Allowing you to respond to incidents quickly.
  • Can set automatic reporting daily/weekly/monthly.
  • Cost effective storing of logs, useful for keeping in with best practise policies in case of an audit.
  • Automatic alerts in case of attack, can set firewalls to automatically block IP ranges.
  • Cloud support for scalability and security.

For Marketing

  • Cost effective and in keeping with best practise policies in case of an audit.

Overview

This page contains important information that every developer working on a BBWT3 project should read through fully. There are links to useful resources through the document to help you understand the way you should write code.

This document is important for all BBWT3 developers - please read and understand it.

Prerequisite Documentation for CSS

CSS can be difficult to understand due to the number of different rules and browser-specific hacks.

However, please take some time to read the following articles to improve your understanding of CSS. Items in bold are useful even for experienced developers;

  • MDN - CSS Guides: a basic introduction to CSS syntax, selectors, values and units.
  • MDN - The box model: a very important concept that affects most elements on the page.
  • MDN - Cascade and inheritance: explains the order of importance in selectors.
  • MDN - Styling boxes: basic guides on styling borders, backgrounds, tables etc.
  • MDN - Pseudo-classes and pseudo-elements: a difficult concept, and some selectors are hard to use in practice, not essential knowledge.
  • MDN - CSS layout: a large collection of guides on layout types - learning this will save you a lot of pain!

Useful Tools

  • CSS Reference: a visual guide to CSS selectors and their properties.
  • Can I use...: check the browser compatibility of CSS features.
  • JSFiddle: in-browser editor for testing out HTML and CSS.

SCSS Pre-Processing

BBWT3 is now using SCSS as its primary CSS pre-processor - you may want to read a quick introduction to it.

SCSS is fully compliant with CSS syntax, compiles into CSS at build-time, and adds these benefits:

  • Visual nesting of selectors - just like HTML.
  • Variables and overrides.
  • Mixins - functions that return CSS with variables.
  • Imports and extends.

Any CSS stylesheet in BBWT3 will automatically be pre-processed if it has a .scss file extension.

Adding and Editing (S)CSS Rules

All CSS rules for the project should be added in the BBWT.Client/src/assets/css directory, in one of the following files:

Important: if you are working on a customer project, rules should be added to project.scss; if you are working on BBWT3 then use BBWT3.scss.

To confirm:

  • project.scss must be used for global overrides in the customer project. This file loads in after BBWT3.scss file and its rules are specific to current project only.
  • If you have CSS improvements that could be relevant for all BBWT projects, then they should be shared with the BBWT3 team, who can move them into the BBWT3.scss file in BBWT3 project.
  • BBWT3.scss contains the BBWT3 global rules. This is also used to override third-party CSS rules. This file shouldn’t be changed within the customer web app because it may get overwritten if we roll out the latest version to other currently active BBWT3 projects.
  • Never make changes directly to 3rd party CSS files either. If we update the component or theme files then your changes will be lost.

New files should only be created for large groups of related rules. This makes it easier for developers to find rules, and allows us to re-use them in other projects in the future. Too many files makes it difficult to search for rules and adds to build time!

Blueberry Group : BBWT3 - Data Upload 

Created by Jason Cozza on Sep 21, 2020

Overview

Implementing a polished and reliable CSV import can be challenging – we might need to handle files with millions of rows, and we also need strong consistent error handling.

BBWT3 includes a strong CSV Import module which meets these goals. The CSV Import module can import CSV files as well as Excel (both XLS and XLSX) files. The number of errors allowed is editable by the developer, and is on a per row basis - meaning multiple errors on a single row count as only 1.

  • Automatically detects errors in the file according to the rule-set devised by the developer.
  • BBWT3 will not import any data if an error is detected at any point within the file.
  • A file will be returned to the user with errors highlighted.

The amount of errors it highlights is user definable.

Test Case

  1. Developer wishes to upload a database of clients to the web app.
  2. Developer goes into Data Importer.
  3. Developer uploads CSV or Excel file.
  4. It is checked for any errors, if found the developer is given a file with the errors highlighted.
  5. Developer is able to amend errors such as missing data.
  6. Developer uploads CSV/Excel file again and is informed through the use of a progress bar how far along file is processed.
  7. Developer now has a complete database imported onto the web app for use.

For Testers

  1. Go to the BBWT3 menu and click on "Feature Demo Pages".
  2. From the drop down menu click on "Data Import".
  3. On the "Employees" tab you can select "Import" to import a file, or on the "Configuration" tab to download a sample file.

For Project Managers

  • Importing large data sets accurately is important.
  • When there is a missing entry and the software you are using to import isn't set up to deal with this, you can end up with completely unusable tables.
  • BBWT3 handles errors the right way, it is set up to have a checking system in place to ensure that the data will not be corrupted and will even report back to you with where errors lie.
  • You have the ability to select how many errors are acceptable before stopping the processing of the file. This will allow you to determine if there is a pattern with the errors, and gives you the opportunity to correct them.
  • You have the ability to start your import from which ever row you choose.
  • There is a strong validation element built into the web application, which can be customised by the developer on an "as needed" basis, but cannot be done by the end user.
  • There is a progress bar that displays how far along the import is and if there are more errors it will cancel itself and give you a pop up message stating that.
  • You have the ability to cancel the import at any time during the processing of the file.
  • Included are sample files so you can understand what will happen if there is a problem and why.
  • Limitation: On importing Excel files you cannot import a file with more than one data sheet.

  • Customise from which row you want to begin importing from. Save time by not having to manually delete rows after an import.
  • Comes with sample files to help understand what issues may arise, and what they look like.

Blueberry Group : BBWT3 - General UI Rules for Developers 

Created by Jason Cozza on Sep 21, 2020

Overview

This page gives a quick recap on the general rules that should be used when coding for BBWT3.

 

  • Always use "email" rather than "e-mail".
  • Use the following capitalisation rules:
    • Title case for page titles, tab page titles, headings,titles of popup dialogs, titles of popup messages, menu labels, titles of grid columns, titles of table columns, button labels. Use http://individed.com/code/to-title-case/ for determining the proper English title case - it's not as simple as capitalising every word. We will be rolling out a tool to help developers with this sometime soon.
    • Sentence case for; labels, checkboxes, drop down menus, lists, radio buttons, text edits, labels for groups of radio buttons and checkboxes, strings in drop down menus and lists, text inside popup messages, text inside progress indicators, links to other pages. Sentence case is the 'normal' use in English prose: only the first word is capitalised, except for proper nouns, acronyms, etc. See http://en.wikipedia.org/wiki/Letter_case for details.
  • We should never use "Click _here_" - we should always use more natural sentences.
  • Use one column layouts; avoid two-column layouts.
