Blueberry Group : BBWT3 - Company Management 

Created by Jason Cozza on Sep 21, 2020

Overview

The Companies feature provides an organisation structure for system users. Users are associated with a company which dictates what features, content and branding are displayed to the user. Security permissions are set via Role Management however and not through Company Management.

  • Company contains:
    • Name.
    • Description.
    • Address.
    • Logo.
    • Offices.
  • Office contains:
    • Address.
  • Ability to add, edit and remove companies.
  • Ability to add, edit and remove offices.

Test Case

  1. Company "X" has opened a new office.
  2. Admin creates new office with relevant details.
  3. Details of new office are available to users.

For Testers

  1. From BBWT3 menu click on "Operational Admin".
  2. From the drop down menu click on "Companies".
  3. Select a company from the "Manage Companies" window.
  4. Offices of selected company will be displayed in the "Offices" window below.

For Project Managers

  • Offers the ability to have multiple companies on the same platform.
  • Depending on what company the user is associated with determines company branding, features and content that they can access.
  • You can easily edit who is associated with a particular company making administration quick and easy.
  • Done through CSS which is a very popular language.
  • To be implemented: Currently there is no ability to associate a user with an office (31-01-19).
  • Bug: Upon uploading a company logo it does not resize it.

For Marketing

  • Ability to have multiple companies on the same platform.
  • The branding a user sees, is easy to set up. Upload a logo for a specific company and alter the colour scheme of the theme used with your web app.

 

  • l time communication when multiple users are on the same page thanks to websockets.

Useful Resources

Please consult the Feedbacks Module page for more detailed information and how to include this module within projects.

Test Case

  1. User has an issue with the web application.
  2. User is able to take a screenshot of the issue and upload it directly without any 3rd party software.
  3. User has the ability to use their voice and have it turned into text via an API instead of typing.
  4. Admin receives the message and is able to investigate and fix the issue quickly.

For Project Managers

  • Quick and easy method of reporting issues.

Blueberry Group : BBWT3 - Logging 

Created by Jason Cozza on Sep 21, 2020

Overview

Logging system uses the 3rd party software called Graylog.

  • Logs current user ID.
  • Logs any impersonation currently in place.
  • Logs any "real user name" defined on current user.

There are 3 logging destinations available by BBWT3:

  • Log error file.
  • Greylog.
  • Database.

File logging should not typically be used in live or test environments, and should be considered for development machines only. BBWT3 principles discourage writing to files on the webserver, and discourage developers connecting to the webserver to read files.

Greylog should be the preferred solution as it can accept application logging in addition to logging from other sources, e.g. database, OS, IIS.

Test Case

  1. Web application is being actively tested by QA team.
  2. A tester causes an error.
  3. This error can be diagnosed due to the logging from Graylog.

Developer Notes

During development dbdoc should not need audit logging in addition to the repository. Any work that goes through the metadata table is already being tracked in the source code repository.
Within Serilog you are able to set the minimum level of logging per source. Within the configuration you need to have the section “Override” inside the “MinimumLevel”. For more information and an example see here.

For Testers

Go to https://graylog.bbconsult.co.uk.

  1. Enter in LDAP username and password.

For Project Managers

  • Able to quickly identify issues. Allowing you to respond to incidents quickly.
  • Can set automatic reporting daily/weekly/monthly.
  • Cost effective storing of logs, useful for keeping in with best practise policies in case of an audit.
  • Automatic alerts in case of attack, can set firewalls to automatically block IP ranges.
  • Cloud support for scalability and security.

For Marketing

  • Cloud support and integration - secure and cost effective.
  • Automatic alerts in case of attack.
  • Cost effective and in keeping with best practise policies in case of an audit.

 

Blueberry Group : BBWT3 - Page Load Timer 

Created by Jason Cozza on Sep 21, 2020

Overview

Page load timer is a feature that displays how long it took in seconds to load a particular page. It displays this value in the top right corner when enabled.

When enabled displays stopwatch icon, with the load time in seconds for currently displayed page.

Test Case

  1. Prior to deployment of web app an admin may wish to check for consistency across the site.
  2. Admin wishes for all pages to be responsive for the user.
  3. Admin goes through links of web app and reviews loading time.
  4. Any pages with longer load times can be tracked down and investigated.
  5. Admin can then make changes such as resizing images.

For Testers

  1. On each loading in the app, there is an icon in the top bar with the loading time:


     
  1. This time is saved in DB, There is a page accessible only for admin to show all loading times:


     
  1. On this page, you are able to filter loading times by Account and page. There is an average time according to these filters:


     
  1. We can deactivate this feature from settings, and from the loading times page:


     
  1. If the user is a “tester”, he can deactivate this feature in menu when he does a right click on the top bar:
     

For Project Managers

  • Having page loading times easily accessible gives a quantifiable value of site responsiveness.
  • You will be able to identify any problem pages, and then be able to attempt to fix issues, to make the user experience more responsive.
  • Having the ability to store all load times within a database that you are able to search and sort through, helps in identifying problems.
  • There is a log of the user and path to which the site was loading on display within the database. This way you can identify who had the issue, as well as what page they were attempting to access.
  • Limitation: Only provides basic search and filtering capability (31-01-19).
  • Limitation: Does not update the load time when filtering search or sorting tables (31-01-19).

For Marketing

  • Able to search for problem pages prior to roll out - enabling a smooth user experience throughout the site.
  • Able to identify through logs with who and where an issue occurred.

Blueberry Group : BBWT3 - Real User Names 

Created by Jason Cozza on Sep 21, 2020

Overview

When a tester or BB person logs into a BBWT3 test system, they often use a test account. For some future features, we need to know the real name / ID of the current user.

  • Upon successful login with a test/demo account user will be prompted to input real info.

This information will be called upon when Reporting a Problem.

Test Case

  1. A tester account can have the role “BBTester”:


     
  1. When the tester logs in for the first time, we ask to fill out with their real data:


     
  1. This data is used when a tester reports a problem:


     
  1. The form is filled with real data. We can change this data in the user’s profile.

For Project Managers

  • Be able to track down testers.
  • When a single account is used by multiple testers trying to find the one who reported an issue could be complex.
  • BBWT3 forces users on test accounts to fill out their real contact information.
  • However a tester must be made aware of the reasons behind this otherwise they may still fill out the details with unidentifiable information.
  • Limitation: Does not check for validation when entering in information (such as checking email address is in valid format).

For Marketing

  • If you have test accounts for multiple users tracking down who reported what can be difficult - with BBWT3 when a new test users logs into the account they will be prompted to enter their real details, resolving this issue.

Blueberry Group : BBWT3 - Report a Problem 

Created by Jason Cozza on Sep 21, 2020

Overview

BBWT3 has an inbuilt Report a Problem feature, that can take the users logged on account details (such as username and email address) and fill them into the template form.

  • Auto fills out user's details (however they still remain editable).

Client side validation (31/01/19)
* Currently does not seem to have any (does not check if email address conforms to standard).
* Unlimited char limit.

  • At request notifications can be setup to be sent via SMS.
  • Automatically generates a report within PTS.

Test Case

  1. User has an issue on the web application.
  2. User finds the "Report a Problem" page.
  3. Fills out form and sends to admin with severity.
  4. Admin is able to address the issue quickly so that other users do not face the same problem.

For Testers

  1. Go to the very bottom of the page, in the footer click on "Report a Problem".
  2. The username and email of the logged in account should automatically be filled in, enter the title of issue.
  3. Write the description of the issue in the "Problem Description" box.
  4. Select the severity of the issue encountered.
  5. Click Submit.

For Project Managers

  • Being able to have feedback on your web application can be useful for many reasons, but primarily for reporting back user issues.
  • BBWT3 includes an easily identifiable Report a Problem feature, so if a user has an issue they can quickly report it.
  • An Admin can set to who the email notification will be sent to.
  • By automatically filling out the users details this saves time and gives the system admins all the information that they need straight away.
  • If a test user is logged in, the details will be that which was entered when they entered their real details at session login. This way you can be sure to be able to contact the original tester if needed, instead of having to ask who was using the account at a specific time or day.
  • There is the option to have a title and a description box to outlie the issue encountered, what is more useful to you will be the ability of the user to state the severity of the issue. Ranging from minor cosmetic issue such as formatting or spelling mistakes, all the way up to complete system failure for multiple users.

For Marketing

  • Decreases the amount of effort a user needs to employ to report back issues with the service - by decreasing the amount of steps to provide feedback, leading to a better experience for all users.
  • Improved user engagement.
  • Improved communication between a user who spots the issue and the person responsible for fixing it - even on test accounts, because of the inclusion of entering real information once logged on.
  • Ability to focus efforts - by having the option for a user to state the level of severity of an issue, this allows for quick and easy prioritisation.

Blueberry Group : BBWT3 - Server Info Dialog 

Created by Jason Cozza on Sep 21, 2020

Overview

Generates static info of server data such as;

  • Server name.
  • Full build number.
  • Server and Client IP's.
  • Operating system.

Test Case

  1. Admin is receiving reports of broken functionality from users.
  2. Admin tracks down that the issues are coming from one particular server.
  3. Admin can quickly check which build version is running to see if it is of an old build.

For Testers

  1. Go to the bottom footer of the page and click on the (I) icon.

For Project Managers

  • Quickly see server information such as IP and name of server.
  • Able to see full build number.
  • Able to see full server OS build directly from browser.

For Marketing

  • BBWT3 displays valuable server information such as; IP address, operating system, server name, version number and server name.
  • All this can be a valuable asset when attempting to track down a problem.
  • For instance you may have multiple servers running the web application, and one of them is not performing correctly. BBWT3 can display the aforementioned data which may lead to resolving the issue.

Blueberry Group : BBWT3 - Version Number Display 

Created by Jason Cozza on Sep 21, 2020

Overview

Displays the build number in the bottom right corner of the footer.

  • Easily visually identifiable.
  • Full build can be seen by accessing Server Info Dialog.

Test Case

  1. User is in a development environment and discovers an issue with BBWT3.
  2. User can check which version of BBWT3 is being used and report the issue.
  3. Developer can then check whether the issue is due to an older build which has now be addressed, or if it is a new issue in a current build.

For Project Managers

  • Useful for keeping track of build's and bugs.
  • It is automatically generated by GitLab.
  • Can also be tracked in PTS.

For Marketing

  • Allows for testers to track changes that have occurred and ensures that there is no confusion; for example, testing on an older build than the current one could lead to wasted time and in turn money.
  • Having the version number there allows for everyone to known if it is the latest build available.
