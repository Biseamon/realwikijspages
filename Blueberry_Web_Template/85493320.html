Blueberry Group : A3:2017 - Sensitive Data Exposure 

Created by Jason Cozza on Sep 22, 2020

Overview
 

Sensitive data exposures may occur when security controls – such as HTTPS – are not implemented correctly, thus leaving a hole for attackers to steal sensitive information such as passwords, payment information, IDs, addresses, and anything else you may have stored which can be of value. Applications should ensure that access be authenticated and data be encrypted. A failure of such may lead to a major privacy violation.

Example

A user in a public place attempts to login to the web application via a login form.

The web page is not secured via HTTPS protocol.

A malicious user on the same network as the user is sniffing for passwords and usernames.

Because the connection to the web application is not secured with HTTPS, the details are sent over the network in clear text.

The malicious user has successfully captured the users login details and can now use them to sign in.

How BBWT3 Mitigates This Vulnerability

  • HTTPS connection.
  • Stores passwords as salted hashes with PBKDF2 encryption.
  • Uses industry standard modern encryption methods.
  • No verbose error messages

     

Connections to and from a web application made from BBWT3 are secured over HTTPS using the TLS protocol. This ensures that any packets set to and from the server are encrypted. This means that even if an attacker is able to intercept these packets, they are useless to them as they are encrypted with the strong encryption offered by TLS.

Passwords are stored securely using the PBKDF2 algorithm to hash passwords instead of storing them in plain text or using obsolete hashing algorithms such as MD5. The passwords before hashing are individually salted - the use of a pseudo-random number is generated and added to the password prior to hashing it - defeating an attackers use of rainbow tables (pre-compiled password hashes).

By doing these things sensitive data is kept secure, significantly limiting the risk of a data breach.

Consider also, the option of encrypting the database files. "Encryption at rest" protects from circumstances where an attacker gains access to the database files without necessarily knowing the database login credentials. 

BBWT3 has the option to limit 'useful messages' to non registered users. By enabling this option we prevent potential attackers from gathering information such as who is registered on the database. This can be done for example when a user enters in a username and incorrect password, instead of saying that the user does or does not exist, a generic message is displayed saying the combination of username and password is invalid.
