Tabs 

 

Overview

This page details how to construct tabs within BBWT3.

 

Use tabs to alternate between attributes belonging to, or strongly associated with, an instance of a particular object. Do not use them to navigate to the management of objects that are only associated with the type of your object.

E.g. Imagine that your "reports" have a set of "parameters" associated with each report, i.e. a parameter "Ordering Company" is a filter for the "Banana Orders" report. The reports also have a set of "report users" that are allowed to view any report, i.e. "John Smith" is allowed to see any of the reports. Note that "report users" are associated with "reports" in general, not one particular report. In this case, your Edit Report page might have a tab for parameters, but it should not have a tab for the report users. Instead, there should be a "Manage Report Users" page.

HangFire 

 

Overview

Hangfire is an open-source framework for running background operations (https://www.hangfire.io/).  It supports a range of scheduling configurations and also runs in a load balanced environment. The implementation in BBWT3 consists of a separate database to store the jobs. Hangfire has its own dashboard to show the status of its jobs, this can be accessed from https://bbwt3-test-1.blueberrycf.com\jobs Not available as of (05-02-19).

  • Manually perform jobs.
  • Run scheduled jobs.
  • Retry functionality on failed job.
  • View scheduled jobs.
  • View past jobs.
  • Search Hangfire database jobs past and present.

Test Case

  1. Admin needs to make backup's of database.
  2. Admin can schedule an export of the database with Hangfire.
  3. Users will not be affected by performance hit as the schedule can be done at night.
  4. Nightly backups can be performed autonomously and be completed by the start of the working day.

For Testers

  1. Go to the BBWT3 menu and click on "Operational Admin".
  2. Click on "System Configuration" from the drop down menu.
  3. Go down to and click on "Hangfire Job List".
  4. From here you can:
    1. View the Schedule (last and next run times).
    2. Enable/Disable the job.
    3. Edit the Schedule.

 

Note that the schedule periods are set using Cron format  (See: Cron)

Developer Notes

Hangfire jobs are created in the database during startup.cs, which needs to have a default schedule defined, there is an example in the BBWT3 code base. However once set up the interval can be overridden using the system configuration menu.

In practice we found that the number of Hangfire connections to the database was disproportionately large to what it was actually doing.  The out of the box default for the number of workers is high; so we have reduced it, this is particularly so in multi-container systems where each container has a full set of workers. So the number of workers you need is less if on one server.  The connections used by Hangfire is about 8-10 with one worker. This seems a lot, but if you  have a full-size DB, the permitted connections is 1000. So the 8-10 is only a problem IF you’re on a micro server.

For Project Managers

  • Having jobs handled in the background saves time and money. Send out mass notifications such as news letters easily; with in built retry mechanism, if there is a failure to complete it will retry after an increasing amount of time.
  • Generate graphs and recurring automated reports.
  • Hangfire helps with database maintenance by performing routine checks such as pruning old users from the database.
  • Has the ability to enqueue - "fire and forget".

For Marketing

  • Hangfire works on both Linux and Microsoft platforms utilising either MySQL or MS SQL.

  • Only allow specific IP addresses to access certain accounts.

  2. An attacker is unable to login to an admin account even with full credentials due to not being on the IP whitelist.

For Project Managers

  • Improving security by blanket denying any access outside of your own internal network.
  • Limitation: Potential issues with remote management. However this can be negated by requiring an admin to first connect to the business VPN before logging into the account.

For Marketing

Password Managers 

 

Overview

BBWT3 supports a variety of the most common password mangers including LastPass and 1Password. Password managers are a valuable way of storing strong user passwords in one place. The passwords stored within the manager have to be accessed by a master password - this enables a user to only have to remember 1 strong password. In turn this keeps all passwords safe as site specific passwords can all be different and be made of an assortment of characters (using a password generator), but the user only needs to remember the 1 master password.

For Project Managers

Session Expiry 

 

Overview

In BBWT3, after a configurable time-out, the system will show a banner message at the top of the screen “You will be logged out in 30s”. This counts down, and at the end of the time the system will redirect to the login page.

For security reasons BBWT3 automatically logs a user out after 10 minutes by default. This can be altered to the desired length depending on the users needs and risk assessment. Session timeout is set in minutes with a minimum of 1 minute allowed for session expiry. The counter is stored and runs down client side.

  • After user has been idle for a designated period of idle time allowed, user will be presented with "You will be logged out in (X)" seconds. It is reset if there is mouse movement.
  • If user does not interact after this message is displayed and the countdown reaches "0", the user will be logged out.

 

(31-01-19)

Currently a user is issued with an access token that does not expire for 14 days. As long as the user does not log out or time out then changes to that users permission will not occur until the access token has expired.

This should be changed to just a few minutes, a user should also be given a refresh token which permits you to refresh the access token without having to log back in again which can be longer - using a sliding window (where the refresh token will be refreshed each time there is user activity on the web application).

 

Test Case

  1. User is in a public place.
  2. User logs into web app.
  3. User leaves computer unattended.
  4. Session expires and user is automatically logged out.
  5. An attacker attempts to take advantage of unattended computer.
  6. Attacker finds that the account is logged out.
  7. Account security maintained.

For Testers

  1. Login as admin and follow instructions for Session Handling to activate session time out and set length.
  2. Login as user and wait for the set length to draw near.
  3. 30 seconds before session expires, a message should appear at the top of the screen.
  4. After message countdown decreases to 0 user will be logged out.

For Project Managers

  • Having a timeout for sessions, improves security for when the user is away from keyboard. Decreasing the likelihood of an unauthorised person gaining access to an account that has been left vulnerable; by not being logged out or the OS locked.
  • Limitation: Can be frustrating for users if session expiry time is set too low.
  • Bug: Currently has issues with decimals (31-01-19 #174154).

For Marketing

  • Having a timeout for sessions, improves security for when the user is away from keyboard in an insecure environment.

 

User Lock Feature 

 

Overview

Users can be locked out through the use of the "Failed Attempts at Password Entry" setting.

If a user incorrectly enters a password the system is able to lock the account depending on the settings that an admin has chosen. For example they are able to lock accounts after “n” number of consecutive failed attempts within a given number of minutes.

  • Prevents brute force attacks.
  • Times out accounts based on settings.
  • Can lock accounts that have had too many failed attempts - requiring an admin to manually unlock them.

Test Case

  1. Attacker who knows a valid admin username attempts to brute force password.
  2. Admin has configured "Failed Attempts at Password Entry" to lock user out until manually reset by another admin.
  3. Attacker is now unable to attempt any other passwords on that account.
  4. Admin will be notified that an account has been locked.
  5. Admin will have to manually unlock the account, and has become aware of an attempted breach as detailed within logs.

For Testers

  1. From the BBWT3 menu click on the "Operational Admin" section.
  2. From the drop down menu click on "System Configuration".
  3. Go down to "Failed Attempts at Password Entry".
  4. Click on the box next to "Users are locked out of their accounts after".
  5. Enter in how many consecutive failed attempts at login that will trigger the lock.
  6. Enter in how many minutes to count the failed logins for.
  7. Select from the box either; how long the account will be locked for, OR if the user will require manual admin changes to unlock their account.

For Project Managers

  • Having timeout and lock account features can help prevent accounts being compromised.
  • A brute force attack could be utilised where a login name is known, such as "admin", but the password is not.
  • The attacker will attempt many different passwords to try and get the correct one, and gain unauthorised access to an account, potentially causing serious harm and disruption.
  • By using the user lock feature, you have the ability to deny access to an account that has had too many failed attempts at login, either for a set amount of time, or permanently until it is manually reviewed by an admin.
  • Issue: If user locks account, can be unlocked by doing password reset without admin intervention.(31-01-19 #177006).

For Marketing

  • Inbuilt brute force protection - accounts that have had too many failed login's are automatically timed our, or if necessary blocked until an admin has unblocked them manually.

 

Maintenance Mode 

 

Overview

This is a feature that allows the system to become unavailable to users whilst maintenance is done.

This would normally be used for system updates but could be used for anything else that requires zero access from users.

In order to use it, a time period for the maintenance needs to be configured. This should be agreed with all the customer and admin users. There is a section of the system configuration page where you can define the start time and end time, a message/warning text plus a check-box to enable/disable the feature. This information is stored in the database.

There are two modes – Basic and External

For Basic mode, the web service is NOT stopped so the user can be redirected to a holding page with a suitable message.

External mode (not implemented yet) is used when the web site needs to be updates. In this instance, the user cannot be redirected to a local page so we would redirect to an external site/page.

External mode does not have a time period – it is effective immediately

How it works

The client needs to know when the maintenance window starts and ends. If it is enabled, then each of the following actions will include the maintenance window info in the returned message;

  •  
    • Page Load/change – each new page change
    • API call – each dbase access from the client
    • Scheduled API call – every 10 mins a call

Once the client knows that the maintenance window it needs to take action.

If we’re within 10 mins of the start of the time period then a message should be displayed across the top of the screen to warn the user. This feature has not yet been implemented.

At the star of the time period, the client will show a DIV over the page with the pre-configured message and an indication of when the time period will end (this updates dynamically every minute).

Being a DIV, the user is unable to access any controls on the page but it should still be visible.

 

When the time period ends, the user is logged out and redirected to the login page.

 

If a user opens the site during maintenance mode the above panel should be displayed automatically and therefore prevent any access.

BUT – at the moment there is a problem where the user is allowed to login and navigate a little bit. So, it looks like we don’t do the check on page load.

API

The API used for the 10 minute check is - /api/settings/maintenancesettings 

Test Case

  1. A planned update is to take place.
  2. The admin/dev can place the website in "Maintenance Mode" so that users are informed when service resumes.
  3. The admin/dev updates the web application as necessary.
  4. Admin/dev takes the site out of "Maintenance Mode" and normal function resumes on the updated application.

For Testers

  1. From the main menu click on "Operational Admin".
  2. Click on "System Configuration" from the drop down menu.
  3. Click on "Maintenance Settings".
  4. Either select "Basic" or "External" and fill out the details as necessary.
  5. Click the checkbox "Enabled".
  6. Go to the bottom of the page and click "Save".

For Project Managers

  • In built maintenance mode.
  • Ability to lock out users while an update is being applied or issue sorted out.
  • Ability to display information on estimated down time for users.

For Marketing

Data Browser 

 

Overview

At the heart of most modern systems is an SQL Database – which to the non-developer is effectively a collection of Excel tables, holding all the data the system works with.  The structure of the database is known as the schema – this is the list of tables and fields, and the relationships between them.

Knowledge of the database schema used by a complex application can be very useful, particularly when a customer has technical staff who might be working directly with data exports or reports themselves.  But documenting the DB schema is often a challenge – the structure evolves rapidly during development, and the tools that exist for DB documentation are complex.

To address this problem, Blueberry has built into BBWT3 a database schema exploration / documentation tool.  This feature is obviously only accessible to administrators (and can be removed from the system if not required).  

This tool allows Blueberry staff to add notes / comments to all tables and fields, and keep this documentation even as the fields and tables change over time.   It also provides a simplified view of the DB structure – hiding unnecessary tables – which make it easier for customer technical staff to understand the system.

But why reinvent the wheel?   By integrating this functionality into BBWT3, it is always to hand, and immediately available.  No need to install SQL tools, nothing to learn.  In our view, this makes all the difference.

  • View existing data.

Write descriptions for data entries.

  • Search through database.
  • Change meta data view.
  • Change validation rules.
  • Export XML file to generate anonymised data with Anonimatron.

  6. Click on "Add View Metadata".
  7. In "Column Width" type 3.
  8. Next to "Column Width" click the drop down and enter "ch" for character.
  9. Click on "Save".
  10. Now when you go to "Demo Pages" => "Rental Car", the "Power" column will be resized to 3 characters width.

Field Widths 

 

Overview

Field widths can be defined through the use of the "metadata" option - there are multiple kinds of metadata planned however currently field width is the only one functioning. Metadata can be associated with a BBWT3 type (e.g. phone numbers) or a table field (eg. Persons.WorkPhone). A BBWT3 type is a form of supplementary description that goes beyond how the database stores the data. The database may say “this column is variable width of unicode characters, up to 254”. A BBWT3 type may add: it’s an email address, reserve a minimum of 30 characters space in the UI for it, and when accepting new data in the UI apply validation using this regular expression, and showing that helpful message if the validation fails. Metadata can be associated with a type at /app/db-column-types . The character allowance does take into account font-sizes used for display. If a page needs to override the metadata, rather than coding it directly on the page, dbdoc supports keyed metadata that will override the default. The key used is the route for the page e.g “/app/rental”. Internally, this is all achieved by:

  • Re-using the automapper element between the DTOs and Model classes, so that any DTO arriving at a page can be traced back to the database fields that supported its creation.
  • The client browser loading this map and associated metadata on initial load of the site.
  • Front-end components refer to this map by … (currently being worked out - 31-01-19)
  • The Type editor and DB Documenter tool use a mechanism whereby changes saved here are turned into GitLab changes. This allows the project manager to merge them into whichever branch they feel is relevant. They’re then no different from any other work performed by developers and subject to the same release processes.
  • The algorithm for how grids make use of the field width metadata is listed in task https://pts.bbconsult.co.uk/taskEditor?id=176533

There’s further work ongoing on how to best display the grid in cases where there is simply insufficient screen-space to meet the metadata requirements.

Page Editor 

 

Overview

Custom applications need ways to guide end-users. However, writing a manual or online help is a significant investment – and if an application is changing rapidly, writing the help can be a never-ending task.

 Blueberry’s view is that help should be delivered in two ways:

  • As tooltips within the application, which give users immediate guidance and examples of what data is expected in a control.
  • As a separate help file which is organised by task or process – so “How to Create a User”.

We see less value in creating screen-by-screen help pages – well, until it can be automatically generated.

As a first step in automating system help documentation, BBWT3 includes an interactive tooltip feature:

  • Developers include (i) icons next to controls.
  • An administrator with the right permission can edit the text for these tooltips interactively directly within the application.
  • The text of the tooltips is automatically sent back to be included in system source code.

Why is this useful? Tooltip text generally needs to be specified by the customer, because it is the customer’s system and they have the domain knowledge.  In previous systems, customers would have to go through and write a long list of the tooltip texts and send to us to include. This was a time-consuming process. By automating this process we save a lot of time, and make it easier to ensure that we have good, strong tooltips in place.
Tooltips are being replaced by a HTML editor, where you can enter a basic HTML editor and create or edit the text for prompts or tooltips.

Test Case

  2. Client admin gets feedback about web application from users.
  3. Users need clarification on a feature.
  4. Admin can create a tooltip icon and description next to the feature.
  5. Client does not need to spend time and money on developers to do this.

User Management 

 

Overview

BBWT3 supports comprehensive user management, with support for Users, Groups and Roles/Permissions.

In practice, the detail of exactly how users are managed depends strongly on the customer application:

  • Some applications have a flat user structure – there’s one set of users, perhaps organised into Groups.
  • Other applications are designed to allow end customers to login and manage their own organisation within the system.
  • In the most complex case, we have a multi-level hierarchical structure of e.g. sales regions or customer offices/sites.

 The core functionality of BBWT3 supports Users, Groups and Companies, but we usually spend time customising and rearranging this code to support exact customer requirements.

Proposed Structure

  • SuperAdmin - Global Administrator, can see everything.
  • SystemAdmin - Customer / System Admin – can see everything except SuperAdmin Role and SuperAdmin Company (based on level).
  • Company Admin - Administrator for a company / organisation – can only see own company users / groups.

Hard-Coded Rules

  •  
    • SuperAdmin can see and edit all user, company records in the system.
    • SystemAdmin can see all user, company records, but cannot edit SuperAdmin user or SuperCompany company.

Changes to the UI

  1.  
    1. SuperAdmin can:
      1. Users screen includes company name in the grid.
      2. Create, edit and delete Global Admins users.
    2. System Admin:
      1. Users screen includes company name in the grid.
      2. The ability to create, edit and delete a company (except SuperCompany).
      3. The ability to create, edit delete users (except SuperAdmin).

         
    3. CompanyAdmin:
      1. Users screen does not include company name in the grid.
      2. Users screen only shows users from same company.
    4. All Accounts:
      1. Users screen should include a list of comma-separated groups.
      2. On user details screen, we have list of groups that this user is in.

User Registration

Prerequisites

User accounts have a status associated with them;

  •  
    • Invited: An invitation email has been sent but the user has not taken required action yet.
    • Unapproved: A system administrator must approve the account before it can be used.
    • Unverified: The account has been approved by an administrator approved but the email address has not yet been verified.
    • Active: The account setup process is complete and it is ready to use.
    • Suspended: The account has been temporarily suspended (by an administrator) and cannot be used.
    • Deleted: The account has been flagged as deleted, and cannot be used.

Invitation Only / Closed

Users with the BBWT3 CompanyAdmin, SystemAdmin, SuperAdmin roles have the ability to generate new user accounts. The first stage is to generate an invitation.

The "+Add" button on the /app/users screen triggers a pop-up window.

This requires the following information, via a form:

  •  
    • Email address.
    • First name.
    • Last name.
    • Company.
    • Role.

For the "Company" field, SystemAdmin and SuperAdmin user are presented with a drop-down list of companies as defined in BBWT3. CompanyAdmin users will not be able to chose the company, accounts created by this role will have the company pre-populated with the same company as CompanyAdmin.

Roles will be presented in a drop-down list:

  •  
    • For SuperAdmin, choices: SuperAdmin, SystemAdmin, CompanyAdmin.
    • For SystemAdmin, choices: SystemAdmin, CompanyAdmin.
    • For CompanyAdmin, choices: CompanyAdmin.

The invited user must now use the supplied URL from BBWT3 to confirm their email address. A token is used to validate the user since they do not have a password yet.

After clicking on the email link, the user will be redirected to BBWT3 and promoted to enter in a password and confirm it. The account status will then be set to "active".

Open

A prospective user visits the registration form in the BBWT3 application, they complete the following mandatory fields:

  •  
    • First Name.
    • Last Name.
    • Company.
    • Password.
    • Confirm password.

The “Register” button will only be enabled if the field validation rules pass.

Actions on "Register":

  • Check that the user does not already exist (by email address).
  • Register user in database with “unverified” status.

Following registration a confirmation message that can be altered in the Technical Admin section will be displayed, e.g:

“You will receive a confirmation email once your application has been approved.”

After which the user will go for approval. There are two options for this stage:

  •  
    • A user with a minimum of CompanyAdmin role can manually change the status of the user account (from “unapproved” to “unverified”), this will be handled via the BBWT3 user management feature.
    • The BBWT3 system automatically approves users and moves the process on to the next stage. This effectively bypasses the approval stage.

       

Actions, on approval:

  •  
    • Set an expiry duration for the access token.
    • Send a confirmation email to the specified address, containing access URL.
    • Send an email to users with the System Administrator role notifying them of pending account request.
  • The invited user must now use the supplied URL to confirm their email address using a secure hash. The URL will open the main homepage of the BBWT3 site logging them in automatically. After which the URL will expire and on further login the user must enter in username and password. The user's status will now be set to "active".

 

User Management Options

  1. Resend invitation: Re-sends the invitation email and resets the expiry timer,
  2. Resend email confirmation: Re-sends the email address confirmation email and resets the expiry timer.
  3. Approve: Sets the account status to “active”. Triggers approval confirmation email.
  4. Password reset: Triggers the reset password process (detailed under self-service task)
  5. Suspend account: Changes account status to “suspended”, disallows login. A failure message must be displayed when the user cannot log in "Your account has been suspended".
  6. Unsuspend account: Changes status back to “active”
  7. Delete account: Changes account status to “deleted”. (We retain the record)
  8. Change email: Allows the registered email address to be changed and then confirmed.

Session Handling 

 

Overview

Sessions are synchronised across open tabs and windows. This  makes the user experience consistent in day to day usage and when troubleshooting user issues, with the added benefit of increased security. This is achieved by a  TypeScript module named CurrentUserWatchService (app/services/current-user-watch.service.ts. This checks every second to see if the login email has changed, it is stored locally and is then overwritten when a new email is used, resulting in a refresh of pages across the different tabs and windows.

  • Will only allow one account to be active in one browser at a time.
  • Logging out on one window or tab will log out and reload the pages on all of them.
  • Attempting to login with more than one account will refresh the pages across tabs/windows with the latest user to login.

Test Case

  1. User is logged in and has an issue with the web application.
  2. Admin goes to user with issue and opens a new tab.
  3. Admin logs in with their credentials.
  4. Admin starts exploring pages and opening tabs.
  5. Admin clicks on previous tab which was originally logged in by the user.
  6. As page has refreshed with latest permissions there is no inconsistency or confusion.

For Testers

  1. Click on the "Operational Admin" drop down box from the BBWT3 menu.
  2. Click on "System Configuration".
  3. Go down to and click on "User Session Settings".
  4. Click "Is Enabled" (if not already enabled).
  5. Set the time in "Idle time (in minutes)" to desired length.
  6. Click "Save" at the bottom of the page.

For Project Managers

  • BBWT3 ensures that only 1 account can be active on a single users web browsing session. Preventing confusion when attempting to complete a task, while not being aware of being logged in simultaneously to more than one account across multiple tabs and windows. Such a case could arise with two different login accounts with differing privileges.
  • This can also cause issue when attempting to troubleshoot, for example with a particular group that is having trouble. If the user is not paying sufficient attention to which account with the particular set of privileges they are investigating, this could cause increased delays in resolving the issue.
  • Provides an additional layer of security in situations where more than one user has access to a particular computer. By having sessions update to the latest user that logged in, it forces a refresh of the pages open across all session instances. Preventing a user with a lower set of privileges gaining access to more restricted information. This coupled with customisable session time outs can be used to improve security in environments where a user has the potential to leave their devices unattended.
  • Limitation: Not being able to be logged into multiple accounts may be frustrating for system admins, who are attempting to troubleshoot user accounts. This can however be bypassed if so desired by using 2 different browsers for the separate accounts.
  • Limitation: Session time out may be frustrating to end users. If it is set too short, it can become tiresome to keep logging in repeatedly. However if it is not enabled or sessions are set too long it can become a security issue. It is up to the individual company and their threat model to determine the acceptable idle session length.

For Marketing

  • Remove confusion when using multiple tabs / windows and diagnosing problems.
  • Improved security in environments where computers are used by multiple users - automatically refreshes pages when new user logs in, preventing unauthorised access to privileged pages.
  • Improved security in environments where computers are left unattended - automatically logs out a user after designated amount of time. This limits the amount of time an attacker would have to gain access to the machine.
