Blueberry Group : BBWT3 - Impersonisation

Created by Jason Cozza on Sep 21, 2020

Overview

Impersonate is a feature to allow an admin user to impersonate another user or group of users. It is often useful to see the restrictions or privileges another user has.

  • Admins are able to view the web application from the perspective of users with varying permissions.

Test Case

  1. A user is having an issue with a feature of the web application.
  2. Instead of relying on fragmented information from user reports, admin has the ability to impersonate the user.
  3. Admin is able to see first hand what the issue the user is facing.
  4. Admin can check if it is unique to a single user, or a group of users with the same permissions.
  5. Admin can take corrective action.

For Testers

  1. From the BBWT3 menu click on "Feature Demo Pages".
  2. Click on "Impersonate".
  3. On this page it will display your current group (by having the appropriate box coloured in, e.g. Admin).
  4. Underneath your current group there will be the option to "Impersonate" other groups of users by clicking on them (note: you cannot impersonate a group with higher privileges than the account you are currently logged in with).
  5. Once you are done impersonating another group of users, return to the "Impersonate" page and at the bottom click "End Impersonate".

For Project Managers

  • The impersonate feature of BBWT3 is useful for checking each group has the correct privileges prior to roll out.
  • Helping with diagnostics when one group of users have the same issue - you can simply impersonate them and see first hand what issue they are experiencing.
  • The biggest strength of having this feature over dummy or test accounts, is that no new users need to be created to check these permissions issues.
  • Enhancing security by not having multiple accounts with high level permissions, that may be protected by weak passwords - or have been forgotten to be deleted after testing.

For Marketing

  • Test users have correct permissions before roll out.
  • Quickly see the problem from the users perspective.
  • No need to login back and forth between accounts.
  • No need for extra testing accounts which could pose a security risk if not deleted, or protected by weak passwords.
